BizOps Configurator Overview

What the heck is this thing?

This is a Single Page App (SPA) to get inputs from the user and from the Dynatrace API, to transform a stock collection of currated dashboards, and to upload via the Dynatrace API

Why?

There's amazing insights that be unlocked just by using stock Dynatrace dashboard tiles. Building dashboards is pretty easy: just drag and drop, populate a few USQL queries, and poof business relevant insights. How can we make it even faster? Share a currated list of dashboards used across our customer base and customize queries with a few clicks and drags. All using the power of the Dynatrace API that's already available.

Structure

We organized dasbhoards into three tiers: tenant/MZ, application, and user journey.

Tenant level overviews and dashboards give you a macro-level view of what's going on in your environment. Your traditional Ops folks probably care about this stuff.

Application overviews and dashboards give a level deeper focusing just on a single application. Your app owners and developers probably care about this stuff.

User Journeys take you yet a level deeper looking at one path through your application; think of what processes are relevant for your application. Your business owners and marketing folks probably care about this stuff. Does your user sign-up, check-out, buy something, build something, post something, get detailed info on products and services, watch training videos, etc? User journeys give you a quick way to define that particular path, whether it's a happy path, a particular promotion, or just a workflow that employees churn through a 100 times aday. Understanding metrics about and being able to filter down to individual sessions gives you the power to know when the journey is working well or the insights to improve it.

BizOps Dashboard IDs

Wait... these dashboard IDs don't look as random as normal Dynatrace dashboard IDs. What's going on here? For BizOps Dashboards we predefine the dashboard ID to make it easier to track the dashboards as a collection. Couldn't that conflict with a dashboards we create the normal way? Yes! but it's only a 1 in 1.2e^24th chance.
Example:

bbbbbbbb-0001-0001-0001-0000000000001
  • bbbbbbbb - Prefix to denote BizOps Dashboards
  • 0001 - Tenant Overview
  • 0001 - Application Overview
  • 0001 - Journey Overview
  • 0000000000001 - Dashboard number

Reserved Dashboard IDs

  • bbbbbbbb-XXXX-0000-0000-000000000000 - Tenant Overview Dashboard
  • bbbbbbbb-XXXX-XXXX-0000-000000000000 - App Overview Dashboard
  • bbbbbbbb-XXXX-0000-0000-ffffffffffff - Tenant Overview Config
  • bbbbbbbb-XXXX-XXXX-0000-ffffffffffff - App Overview Config
  • bbbbbbbb-XXXX-XXXX-XXXX-ffffffffffff - Journey Config

GitOps

We're using a GitOps approach to maintaining these dashboards. Owners of the dashboards have their own public repos with the JSONs. The configurator just pulls from those repos, transmogrifies, and uploads to your tenant.

  • a tag in your post with @Lucas.H, so I know it's out there

Privacy

We use Dynatrace to monitor this application. Here's what we're capturing:

  • Session Replay - we are masking your tokens, but capturing everything else so we can understand how you're using the tool and why it may have broken.
  • Your tenant URL - this allows us to understand where you're putting the dashboards
  • Username - We grab your username from the authentication response using your token
  • Number of BizOps Dashboards deployed - We want to know how much you're using the tool
  • Your IP / GeoLocation - To understand how your ISP / loction affects performance of the tool

NOTE: If you are sending a DO NOT TRACK cookie, we don't capture any of the above, just anonymous usage. Your use of the tool indicates your consent for this tracking.
